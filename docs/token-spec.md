# Chat9 Token Spec

## Status

This document is the normative Phase 1 token contract for the public `chat9-sdks` repository.

Phase 1 covers Python, Go, and PHP only. SDK code is an implementation of this spec. If implementation and this document diverge, this document wins.

Any change that alters token bytes, verification outcome, or error-code behavior is a breaking change.

## Public API

All Phase 1 SDKs expose:

- `generateToken({ secret, user, options? }) -> string`
- `verifyToken(token, secret) -> payload`

`options`:

- `ttl`: integer seconds, default `300`, valid range `60..3600`
- `algorithm`: optional, must be exactly `"sha256"`

If `algorithm` is present and not exactly `"sha256"`, the SDK must return `INVALID_FIELD`.

No public `now` parameter is allowed. Tests may freeze time internally.

## Token Format

The token format is exactly:

```text
<base64(payload_json_utf8)>.<hmac_sha256_hex(base64_segment)>
```

Rules:

- payload segment uses standard Base64
- Base64 must be padded
- signature input is exactly the Base64 payload segment bytes
- signature algorithm is HMAC-SHA256
- signature output is lowercase hexadecimal

## Payload Contract

Generated payloads must be JSON objects.

Required generated fields:

- `user_id`: required, non-empty, non-whitespace string
- `exp`: generated by the SDK as an integer Unix timestamp in UTC seconds

Allowed optional fields:

- `email`
- `locale`
- `timezone`
- `custom_attrs`

Unknown fields:

- are allowed
- are preserved
- participate in canonical serialization
- must still leave the payload as a JSON object with valid core fields

Compatibility fields such as `tenant_id` are supported for backend interoperability but are not core required fields.

## Canonicalization

### JSON

Canonical JSON serialization rules:

- the payload must serialize as a JSON object only
- UTF-8 encoding only
- no insignificant whitespace
- omit absent fields; do not inject `null`
- preserve input string values exactly after validation
- no trimming
- no case normalization
- no Unicode normalization
- sort object keys lexicographically at every object level
- nested compatibility objects must also be lexicographically sorted
- `custom_attrs` keys must be lexicographically sorted
- preserve array order exactly if arrays appear in compatibility fields

### Base64

Base64 requirements:

- standard Base64 only
- padded Base64 only
- malformed or non-canonical Base64 must be rejected as `INVALID_TOKEN_FORMAT`

## Time Semantics

- `exp` is an integer Unix timestamp in whole UTC seconds
- `generateToken` computes `exp = now + ttl`
- `verifyToken` compares against current UTC Unix seconds
- no clock skew tolerance is allowed in Phase 1
- expired means `now >= exp`
- milliseconds are never used in generation or verification

## Validation Rules

### Secret

- must be a string
- must not be empty
- no trimming before validation
- non-string and empty string both map to `MISSING_SECRET`

### `user_id`

- must be a string
- must not be empty
- must not be whitespace-only
- non-string, empty, and whitespace-only all map to `MISSING_USER_ID`

### `ttl`

- must be an integer
- must be within `60..3600`
- invalid type or out-of-range value maps to `INVALID_TTL`

### `email`

- must be a string
- must contain exactly one `@`
- local part must be non-empty
- domain must contain at least one `.`
- domain labels must be non-empty
- no spaces
- failures map to `INVALID_FIELD`

### `locale`

- must be a string
- must match `^[A-Za-z]{2,3}(-[A-Za-z]{4})?(-([A-Za-z]{2}|[0-9]{3}))?$`
- failures map to `INVALID_FIELD`

### `timezone`

- must be a string
- must match `^[A-Za-z_]+(?:/[A-Za-z0-9_+\\-]+)+$`
- no OS or runtime timezone database lookup is allowed in Phase 1
- failures map to `INVALID_FIELD`

### `custom_attrs`

- must be an object or map if present
- maximum 20 keys
- keys must be strings
- values must be strings only
- each value length must be `<= 256`
- overflow maps to `CUSTOM_ATTRS_OVERFLOW`
- wrong type or wrong shape maps to `INVALID_FIELD`

## Verification Rules

`verifyToken` must apply checks in the following order:

1. Require exactly one `.` separator producing exactly two non-empty segments
2. Verify the HMAC-SHA256 signature using constant-time comparison — any structural error in the payload must not be observable before authentication succeeds
3. Require canonical padded standard Base64 in the payload segment
4. Decode payload as UTF-8 JSON
5. Reject duplicate object keys as `INVALID_TOKEN_FORMAT`
6. Require the payload to be a JSON object
7. Require `exp` to be an integer JSON number only; reject if float, string, boolean, or null
8. Validate optional fields if present
9. Reject expired tokens when `now >= exp`
10. Preserve unknown fields on success

Signature verification must precede all payload content checks so that an attacker cannot distinguish a structurally malformed payload from a bad signature without possessing the secret.

Malformed token shape, bad Base64, invalid JSON, duplicate keys, non-object payload, and invalid `exp` type are all `INVALID_TOKEN_FORMAT`.

## Error Taxonomy

All Phase 1 SDKs must expose a typed `Chat9Error` with:

- `code`
- `message`

Build or input errors:

- `MISSING_SECRET`
- `MISSING_USER_ID`
- `INVALID_FIELD`
- `INVALID_TTL`
- `CUSTOM_ATTRS_OVERFLOW`

Verification errors:

- `INVALID_TOKEN_FORMAT`
- `INVALID_SIGNATURE`
- `EXPIRED_TOKEN`

Error mapping:

- malformed structure, bad Base64, invalid JSON, duplicate keys, non-object payload, invalid `exp` type -> `INVALID_TOKEN_FORMAT`
- signature mismatch -> `INVALID_SIGNATURE`
- expired token -> `EXPIRED_TOKEN`

## Canonical Examples

### Example 1: Minimal payload

Input:

- `secret = "secret_minimal_123"`
- `user = {"user_id":"user-123"}`
- `now = 1700000000`
- `ttl = 300`

```json
{"exp":1700000300,"user_id":"user-123"}
```

