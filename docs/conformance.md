# Chat9 Conformance

## Purpose

This document defines the Phase 1 conformance obligations for Python, Go, and PHP in the public `chat9-sdks` repository.

`docs/token-spec.md` is normative. This document describes how implementations prove compliance.

## Phase 1 Deliverables

Phase 1 is complete only when all of the following exist:

- `docs/token-spec.md`
- `docs/conformance.md`
- `docs/compatibility.md`
- `docs/release-process.md`
- `fixtures/generate_cases.json`
- `fixtures/verify_cases.json`
- `fixtures/golden_tokens.json`

Phase 1 repository structure must contain only:

- `python/`
- `go/`
- `php/`
- `docs/`
- `fixtures/`
- `.github/workflows/`

Do not scaffold `node/`, `java/`, or `ruby/` directories during Phase 1.

## Time Rules

Fixture time semantics are mandatory:

- fixture `now` values are whole UTC seconds only
- milliseconds are never used in fixtures
- milliseconds are never used in generation
- milliseconds are never used in verification

## Shared Test Obligations

Each SDK must pass the same shared conformance data.

### Generation fixtures

Each SDK must:

- generate the expected payload for every valid generation case
- generate the expected token bytes for every valid generation case
- return the expected error code for every invalid generation case

### Verification fixtures

Each SDK must:

- verify the expected payload for every valid verification case
- return the expected error code for every invalid verification case

### Cross-language parity

Each SDK must:

- verify golden tokens generated by the other two Phase 1 SDKs
- produce byte-identical tokens for frozen fixture inputs
- preserve unknown compatibility fields on successful verification

## Minimum Required Case Coverage

The fixture suite must cover, at minimum:

- valid minimal payload
- valid full payload
- valid compatibility-field payload
- missing secret
- missing `user_id`
- invalid `email`
- invalid `locale`
- invalid `timezone`
- invalid `custom_attrs`
- `custom_attrs` overflow
- ttl below range
- ttl above range
- invalid algorithm
- malformed token format
- bad Base64 payload
- invalid JSON payload
- duplicate JSON object keys
- non-object payload
- missing `exp`
- invalid `exp` type
- tampered payload
- tampered signature
- expired token

## Golden Vectors

`fixtures/golden_tokens.json` is a compatibility contract.

It must include committed vectors for:

- one minimal valid token
- one full valid token
- one compatibility-field token
- one expired token
- one tampered-signature token
- one malformed-format token

Golden vectors must not drift silently. Any intentional drift is a breaking change and requires explicit review.

## CI Gates

A pull request cannot merge unless:

- docs are updated for any contract change
- Python tests pass
- Go tests pass
- PHP tests pass
- all three SDKs pass shared generation fixtures
- all three SDKs pass shared verification fixtures
- all three SDKs verify each other’s golden vectors
- no golden token drift occurs without breaking-change review

## Versioning

Use semantic versioning from day one.

Any change that alters:

- token bytes
- accepted or rejected fixture behavior
- error-code behavior

is a breaking change and must be released as a semver-major change.
